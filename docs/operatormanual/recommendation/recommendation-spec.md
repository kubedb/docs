# Recommendation Spec & Status

A `Recommendation` resource contains the specification and status of a recommendation generated by KubeDB Ops-manager. This document describes the key fields in both the spec and status sections.

## Recommendation Spec

The `Recommendation` spec defines the desired state and metadata for the recommendation.

### Key Fields

**description** (string, optional)
- Specifies the reason why this recommendation is generated.
- Example: `"Recommending TLS certificate rotation, elastic-client-cert Certificate is going to be expire on 2025-02-27 12:04:43 +0000 UTC"`

**target** (TypedLocalObjectReference)
- Specifies the APIGroup, Kind & Name of the target resource for which the recommendation is generated.
- Example:
  ```yaml
  target:
    apiGroup: kubedb.com
    kind: Elasticsearch
    name: elastic
  ```

**operation** (RawExtension)
- Holds a Kubernetes object YAML which will be applied when this recommendation is executed.
- Must be a valid Kubernetes resource YAML containing `apiVersion`, `kind`, and `metadata` fields.
- Example:
  ```yaml
  operation:
    apiVersion: ops.kubedb.com/v1alpha1
    kind: ElasticsearchOpsRequest
    metadata:
      name: rotate-tls
      namespace: es
    spec:
      databaseRef:
        name: elastic
      tls:
        rotateCertificates: true
      type: ReconfigureTLS
  ```

**recommender** (ObjectReference)
- Holds the name and namespace of the component which generated this recommendation.
- Typically set to `kubedb-ops-manager`.

**deadline** (metav1.Time, optional)
- The recommendation will be executed within the given deadline.
- To maintain deadline, parallelism can be compromised.

**requireExplicitApproval** (boolean, optional)
- If set to `true`, the Recommendation must be approved manually.
- Recommendation won't be executed without manual approval and any kind of ApprovalPolicy will be ignored.

**rules** (OperationPhaseRules)
- Defines the rules to identify successful, progressing, or failed execution of the operation.
- Contains three CEL (Common Expression Language) expressions.

**backoffLimit** (int32, optional)
- Specifies the number of retries before marking this recommendation as failed.
- Default: 5
- Range: 0-10. If set to 0, the operation will be tried only once.

**vulnerabilityReport** (VulnerabilityReport, optional)
- Specifies any vulnerability report like CVE fixed information.
- Includes status, message, fixed CVEs, and known CVEs.

## Operation Phase Rules

The `OperationPhaseRules` contains three identification rules:

**success** (string, required)
- Rule to identify successful execution of the operation.
- Example: `has(self.status.phase) && self.status.phase == 'Successful'`

**inProgress** (string, required)
- Rule to identify that the applied operation is progressing.
- Example: `has(self.status.phase) && self.status.phase == 'Progressing'`

**failed** (string, required)
- Rule to identify that the applied operation has failed.
- Example: `has(self.status.phase) && self.status.phase == 'Failed'`

The rules use `self` prefix to reference fields in the Operation object:
- `.status.phase` → `self.status.phase`
- `.status.observedGeneration` → `self.status.observedGeneration`

Rules can be any valid CEL expression. Ref: https://github.com/google/cel-spec

## Recommendation Status

The `Recommendation` status reflects the observed state of the recommendation.

### Key Fields

**approvalStatus** (ApprovalStatus)
- Specifies the approval status of the Recommendation.
- Possible values: `Pending`, `Approved`, `Rejected`
- **Pending**: Recommendation is yet to be approved or rejected
- **Approved**: Recommendation is permitted to execute
- **Rejected**: Recommendation is rejected and never will be executed

**phase** (RecommendationPhase)
- Specifies the current phase of the recommendation.
- Possible values: `Pending`, `Skipped`, `Waiting`, `InProgress`, `Succeeded`, `Failed`

| Phase | Description |
|-------|-------------|
| Pending | Recommendation misses at least one pre-requisite for executing the operation. User action may be needed. |
| Skipped | Operation is skipped because of Rejection approval status. |
| Waiting | Recommendation is waiting for the MaintenanceWindow to execute or waiting for other recommendations to complete for parallelism. |
| InProgress | Operation execution has successfully started and waiting for its final status. |
| Succeeded | Operation has been successfully executed. |
| Failed | Operation execution has not completed successfully and encountered an error. |

**reason** (string)
- A message indicating details about the recommendation's current phase.
- Default: `WaitingForApproval`

**reviewer** (Subject, optional)
- Specifies the reviewer's details (who approved/rejected the recommendation).
- Contains: Kind, APIGroup, Name, Namespace

**comments** (string, optional)
- Specifies the reviewer's comment.

**reviewTimestamp** (metav1.Time, optional)
- Contains the review timestamp.

**approvedWindow** (ApprovedWindow, optional)
- Specifies the time window configuration for the recommendation execution.
- See [ApprovedWindow](#approvedwindow) section below.

**parallelism** (Parallelism)
- Imposes restrictions to recommendation execution.
- Possible values: `Namespace`, `Target`, `TargetAndNamespace`
- Default: `Namespace`

| Value | Description |
|-------|-------------|
| Namespace | Only one recommendation can be executed at a time in a namespace. |
| Target | Only one recommendation for a given target can be executed at a time. |
| TargetAndNamespace | Only one recommendation for a given target can be executed at a time in a namespace. |

**outdated** (boolean)
- Indicates whether the recommendation is outdated or not.
- If `true`, the recommendation will not be executed.
- This indicates that after generating the recommendation, the targeted resource changed such that the recommendation can't be executed anymore.
- Default: `false`

**conditions** ([]Condition)
- Conditions applied to the recommendation.

**createdOperationRef** (LocalObjectReference, optional)
- Holds the created operation name.

**failedAttempt** (int32)
- Holds the number of times the operation has failed.
- Default: 0

**observedGeneration** (int64, optional)
- The most recent generation observed for this resource.

## ApprovedWindow

The `ApprovedWindow` specifies when and how a recommendation should be executed.

**window** (WindowType)
- Defines the approved window type.
- Possible values: `Immediate`, `NextAvailable`, `SpecificDates`

| Value | Description |
|-------|-------------|
| Immediate | Recommendation will be executed immediately. |
| NextAvailable | Recommendation will be executed in the next available maintenance window. |
| SpecificDates | Recommendation will be executed in the given dates. |

**maintenanceWindow** (TypedObjectReference, optional)
- Holds the reference of the MaintenanceWindow resource.
- Used when a specific maintenance window is selected.

**dates** ([]DateWindow, optional)
- Holds a list of DateWindow when the recommendation is permitted to execute.
- Only used for window type `SpecificDates`.
- Each DateWindow contains Start and End timestamps in UTC format.

### ApprovedWindow Scenarios

**Scenario 1: Default Maintenance Window**
- User provides nothing and default MaintenanceWindow will be used.
- If no default window is found, recommendation stays in `Pending` state.
- Priority: NamespaceScoped > ClusterScoped

**Scenario 2: Immediate Execution**
- User provides window type `Immediate`.
- OpsRequest will be created immediately.

**Scenario 3: Specific Maintenance Window**
- User provides a specific MaintenanceWindow reference.
- That window will be used or an error will be thrown if not found.

**Scenario 4: Next Available Window**
- User provides window type `NextAvailable`.
- First namespace-scoped available window will be used.
- If not found, next available ClusterMaintenanceWindow will be used.
- If still not found, recommendation stays in `Pending` state.

**Scenario 5: Specific Dates**
- User provides window type `SpecificDates`.
- User must provide at least one DateWindow in the dates field.
- Otherwise, controller will throw an error.

## Example Recommendation

```yaml
apiVersion: supervisor.appscode.com/v1alpha1
kind: Recommendation
metadata:
  name: elastic-x-elasticsearch-x-rotate-tls-6ujvez
  namespace: es
spec:
  description: Recommending TLS certificate rotation, elastic-client-cert Certificate is going to be expire on 2025-02-27 12:04:43 +0000 UTC
  target:
    apiGroup: kubedb.com
    kind: Elasticsearch
    name: elastic
  operation:
    apiVersion: ops.kubedb.com/v1alpha1
    kind: ElasticsearchOpsRequest
    metadata:
      name: rotate-tls
      namespace: es
    spec:
      databaseRef:
        name: elastic
      tls:
        rotateCertificates: true
      type: ReconfigureTLS
  recommender:
    name: kubedb-ops-manager
  deadline: "2025-02-27T11:59:43Z"
  rules:
    success: has(self.status.phase) && self.status.phase == 'Successful'
    inProgress: has(self.status.phase) && self.status.phase == 'Progressing'
    failed: has(self.status.phase) && self.status.phase == 'Failed'
  backoffLimit: 5
status:
  approvalStatus: Pending
  phase: Pending
  reason: WaitingForApproval
  parallelism: Namespace
  outdated: false
  failedAttempt: 0
```

For more details on how to approve/reject recommendations, see [Rotate TLS Recommendation](/docs/operatormanual/recommendation/rotate-tls-recommendation.md).
